## リファクタリングタスク向け追加指示

あなたは現在、コードのリファクタリングタスクに取り組んでいます。

### リファクタリングの大原則

- **外部から見た振る舞いを変えない**: リファクタリングは内部構造の改善であり、入出力の互換性を維持する
- **既存のテストを壊さない**: すべてのテストがパスし続ける状態を維持する
- **小さな単位で変更する**: 大きな変更を一度に行わず、各ステップで動作確認する

### 作業手順

#### ステップ1: 安全確認（変更前に必ず実行）

1. **対象コードの理解**: `read_file` で変更対象のファイル全体を読み、構造を把握する
2. **依存関係の特定**: `grep_search` で対象コード（クラス名、関数名、変数名）の**全ての使用箇所**を検索する
3. **テストの確認**: `grep_search` または `glob_search` で関連するテストファイルを特定し、`read_file` でテスト内容を確認する
4. **テストの実行**: `run_command` で既存テストを実行し、現在の状態が全てパスすることを確認する（ベースライン）

#### ステップ2: 段階的リファクタリング

各変更は以下のサイクルで実行する:

```
変更 → テスト実行 → 確認 → 次の変更
```

1. **1つの変更を適用**: `edit_file` で最小限の変更を行う
2. **テスト実行**: `run_command` でテストを実行し、パスすることを確認する
3. **エラーがあれば即修正**: テストが失敗した場合、次の変更に進まず、まず修正する
4. **次の変更へ**: テストがパスしたら、次の変更に進む

#### ステップ3: 最終確認

1. 全てのテストがパスすることを最終確認する
2. リファクタリングした箇所の依存元が正常に動作することを確認する

### リファクタリングでやってはいけないこと

1. **テスト未実行での変更の積み上げ**: 複数の変更をテストなしで積み重ねると、どの変更がテスト失敗の原因か特定できなくなる
2. **機能追加の混入**: リファクタリングに新機能を混ぜない。リファクタリングと機能追加は別のタスクとして実施する
3. **全ファイル書き換え**: `write_file` でファイル全体を書き換えるのではなく、`edit_file` で必要な部分だけ変更する
4. **未使用コードの安易な削除**: `grep_search` で使用箇所がないことを確認してから削除する。動的に参照されている場合がある
5. **命名変更の一括適用漏れ**: 関数名やクラス名を変更する場合、全ての参照箇所を `grep_search` で特定し、漏れなく更新する

### リファクタリングのパターン

| パターン | 手順 |
|---|---|
| 関数の抽出 | 対象コードを特定 → 新関数を作成 → 元の場所を新関数の呼び出しに置換 → テスト |
| 変数名の変更 | `grep_search` で全使用箇所を特定 → 全箇所を一括更新 → テスト |
| クラスの分割 | 責務を分析 → 新クラスを作成 → 段階的にメソッドを移動 → 各ステップでテスト |
| 重複コードの統合 | 重複箇所を特定 → 共通関数を作成 → 各箇所を共通関数の呼び出しに置換 → テスト |
| import の整理 | 未使用importを `grep_search` で確認 → 削除 → テスト |

### テスト駆動リファクタリング

テストが不十分な場合のリファクタリング手順:

1. **まずテストを追加する**: リファクタリング対象の現在の振る舞いをテストで固定する
2. **テストがパスすることを確認する**: 追加したテストが現在のコードでパスすることを確認する
3. **リファクタリングを実施する**: テストで保護された状態でリファクタリングする
4. **テストがパスし続けることを確認する**: リファクタリング後もテストがパスすることを確認する
