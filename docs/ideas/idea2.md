# 要求仕様書: CLI AI Agent（Claude Code相当）

## 1. プロダクト概要

### 1.1 目的

Claude Codeと同等の機能を持つCLI AIエージェントを自前で構築する。
ソフトウェア開発の全工程（設計・実装・テスト・デバッグ・リファクタリング）をターミナル上で自律的に支援・実行する。

### 1.2 コンセプト

- 「ターミナルに住む自律型AI開発者」
- ユーザーは自然言語で指示を出すだけ
- エージェントがコードの理解・生成・修正・検証を自律的に実行
- 必要に応じてユーザーに確認を求める

### 1.3 技術スタック

| 項目 | 技術 |
|------|------|
| 言語 | Python 3.12+ |
| フレームワーク | LangChain / LangGraph (v1.0以上) |
| LLMプロバイダ | OpenAI (GPT5-nano) / Google Gemini (2.5 Pro, 2.5 Flash) |
| パッケージ管理 | uv |
| テスト | pytest |
| リンター | ruff |
| 型チェック | mypy |

## 2. スコープ

### 2.1 対象範囲

- 対話型CLIインターフェース
- コード生成・修正・リファクタリング
- ファイルの読み書き・検索
- シェルコマンド実行
- Git操作（status, diff, commit, branch, etc.）
- テスト実行・結果解析
- コードベース全体の理解と横断検索
- タスクの自律的な分解と実行
- エラーの自動検知と修正

### 2.2 対象外（将来拡張）

- GUI / Web UI
- IDE統合（VSCode拡張等）
- リアルタイム共同編集
- マルチエージェント協調

## 3. システムアーキテクチャ

### 3.1 全体構成

```
User (Terminal)
    |
    v
+------------------+
| CLI Interface    |  -- Rich/Textual による表示制御
+------------------+
    |
    v
+------------------+
| Agent Core       |  -- LangGraph ステートマシン
|  ├ Planner       |  -- タスク分解・計画立案
|  ├ Executor      |  -- ツール呼び出し・実行
|  └ Critic        |  -- 結果評価・再試行判断
+------------------+
    |
    v
+------------------+
| LLM Router       |  -- プロバイダ切替・フォールバック
|  ├ OpenAI        |
|  └ Gemini        |
+------------------+
    |
    v
+------------------+
| Tool Registry    |  -- ツール管理・スキーマ定義
|  ├ File Tools    |
|  ├ Shell Tools   |
|  ├ Git Tools     |
|  ├ Search Tools  |
|  └ Test Tools    |
+------------------+
    |
    v
+------------------+
| Context Manager  |  -- コンテキスト管理・圧縮
|  ├ Memory Store  |
|  └ Indexer       |
+------------------+
```

### 3.2 コンポーネント詳細

#### CLI Interface

- 対話型REPL（Read-Eval-Print Loop）
- ストリーミング出力（トークン単位のリアルタイム表示）
- Markdownレンダリング（コードブロック、テーブル等）
- 実行中のスピナー/プログレス表示
- 入力履歴・補完

#### Agent Core（LangGraph ステートマシン）

エージェントの思考ループを状態遷移で管理する:

```
START → Plan → Execute → Observe → Evaluate → [Complete | Retry | Replan]
```

- **Planner**: ユーザーの指示をサブタスクに分解し、実行順序を決定
- **Executor**: 計画に従いツールを呼び出し、結果を収集
- **Critic**: 実行結果を評価し、成功/失敗/再試行を判断

最大ループ回数を設定し、無限ループを防止する。

#### LLM Router

- 複数プロバイダの統一インターフェース
- モデル選択: タスク種別に応じた自動選択（高精度タスク→GPT-4o/Gemini 2.5 Pro、軽量タスク→GPT-4o-mini/Gemini 2.5 Flash）
- フォールバック: プロバイダ障害時の自動切替（OpenAI → Gemini）
- レートリミット管理・リトライ制御
- トークン使用量の追跡・表示

#### Tool Registry

LangChainのTool抽象を使い、統一的なインターフェースでツールを管理する。
各ツールはJSON Schemaで入出力を定義し、LLMが適切に呼び出せるようにする。

#### Context Manager

- 会話履歴の管理と圧縮（古い履歴を要約してトークン節約）
- プロジェクトファイルのインデックス化（ファイルツリー、シンボル一覧）
- 関連コードの自動取得（ユーザーの質問に関連するファイルを特定）

## 4. 機能要件

### 4.1 CLIコマンド

```bash
# 対話モード（デフォルト）
myagent

# ワンショット実行
myagent "このバグを修正して"

# 設定
myagent config set model openai/gpt-4o
myagent config set model gemini/gemini-2.5-pro
myagent config list

# ヘルプ
myagent --help
```

### 4.2 ファイル操作ツール

| ツール名 | 機能 |
|----------|------|
| `read_file` | ファイル読み込み（行番号指定、範囲指定対応） |
| `write_file` | ファイル新規作成 |
| `edit_file` | 既存ファイルの部分編集（差分ベース） |
| `list_directory` | ディレクトリ内容一覧 |
| `glob_search` | パターンによるファイル検索 |
| `grep_search` | 正規表現によるコード内容検索 |

### 4.3 シェル実行ツール

| ツール名 | 機能 |
|----------|------|
| `run_command` | シェルコマンド実行（stdout/stderr取得） |

- タイムアウト制御（デフォルト120秒、設定可能）
- 危険コマンドの検知と確認プロンプト
- 作業ディレクトリの追跡

### 4.4 Git操作ツール

| ツール名 | 機能 |
|----------|------|
| `git_status` | リポジトリ状態確認 |
| `git_diff` | 変更差分表示 |
| `git_log` | コミット履歴表示 |
| `git_commit` | コミット作成 |
| `git_branch` | ブランチ操作 |
| `git_checkout` | ブランチ切替・ファイル復元 |

### 4.5 テスト実行ツール

| ツール名 | 機能 |
|----------|------|
| `run_tests` | pytest実行（指定ファイル/関数単位） |
| `parse_test_results` | テスト結果の構造化解析 |

### 4.6 コードベース理解

- プロジェクト初回ロード時にファイルツリーを自動取得
- ユーザーの質問に応じて関連ファイルを検索・読み込み
- import/依存関係の追跡
- シンボル定義の横断検索（関数、クラス、変数）

### 4.7 ユーザー確認フロー

以下の操作は実行前にユーザーの承認を求める:

- ファイルの書き込み・編集（差分を表示してから確認）
- 破壊的Git操作（force push, reset --hard, branch -D等）
- 外部への通信を伴う操作
- 長時間実行が予想されるコマンド

確認レベルは設定で調整可能:
- `strict`: 全操作で確認
- `normal`: 書き込み・破壊操作のみ確認（デフォルト）
- `autonomous`: 確認なし（上級者向け）

## 5. 非機能要件

### 5.1 パフォーマンス

- ストリーミング応答: 最初のトークンが1秒以内に表示開始
- ツール実行: 各ツール呼び出しは個別にタイムアウト制御
- 並列実行: 独立したツール呼び出しは並列で実行可能

### 5.2 信頼性

- LLM APIエラー時のリトライ（指数バックオフ、最大3回）
- プロバイダ間フォールバック
- 自律ループの最大ステップ数制限（デフォルト25）
- セッション状態の永続化と復元

### 5.3 セキュリティ

- **コマンド実行制限**:
  - 危険コマンドのブロックリスト（`rm -rf /`, `mkfs`, `dd` 等）
  - 実行ディレクトリをプロジェクトルートに制限
- **APIキー管理**:
  - 環境変数または設定ファイル（.env）から読み込み
  - キーをログに出力しない
- **ファイルアクセス制限**:
  - プロジェクトディレクトリ外への書き込み禁止
  - .env、credentials等の機密ファイルの読み取り時に警告

### 5.4 ログ・オブザーバビリティ

- 全エージェント思考ログをJSON形式で保存
- ツール呼び出し履歴（入力・出力・所要時間）
- トークン使用量・コストの追跡
- ログレベル設定（DEBUG / INFO / WARNING / ERROR）

### 5.5 設定管理

設定ファイル（`~/.myagent/config.toml`）で管理:

```toml
[llm]
default_provider = "openai"
default_model = "gpt5-nano"
fallback_provider = "gemini"
fallback_model = "gemini-2.5-pro"
temperature = 0.0
max_tokens = 8192

[agent]
max_iterations = 25
confirmation_level = "normal"  # strict / normal / autonomous

[shell]
timeout = 120
blocked_commands = ["rm -rf /", "mkfs", "dd"]

[logging]
level = "INFO"
directory = "~/.myagent/logs"
```

## 6. プロンプト設計

### 6.1 システムプロンプト方針

- エージェントの役割・能力・制約を明確に定義
- 利用可能なツールとその使い分けルールを記述
- 「推測せずツールで確認する」原則を徹底
- 安全性ルール（確認フロー、禁止操作）を組み込み

### 6.2 役割別プロンプト

| 役割 | 方針 |
|------|------|
| Planner | タスクを具体的なステップに分解。不足情報があればユーザーに質問。 |
| Executor | 計画に忠実にツールを実行。結果を正確に記録。 |
| Critic | 実行結果を検証。期待と異なればリプラン or リトライを判断。 |

---

## 7. コンテキスト管理戦略

### 7.1 コンテキストウィンドウ管理

- トークン使用量を常に追跡
- 上限の80%に達したら古い会話を要約して圧縮
- ツール出力が大きい場合はトランケーション

### 7.2 プロジェクトインデックス

- 初回起動時にファイルツリーとシンボル一覧を生成
- `.gitignore` に従って除外
- インデックスをキャッシュし、ファイル変更時に差分更新

### 7.3 関連コード取得

- ユーザーの質問からキーワードを抽出
- grep/glob でファイルを特定
- 必要最小限のコードをコンテキストに注入

---

## 8. エラーハンドリング

| エラー種別 | 対策 |
|-----------|------|
| LLM APIエラー（429, 500等） | 指数バックオフリトライ → フォールバックプロバイダ |
| ツール実行エラー | エラー内容をLLMに渡し、修正策を再計画 |
| 無限ループ検知 | 同一ツール呼び出しの繰り返しを検知、最大ステップで強制終了 |
| パース失敗 | LLM出力が不正な場合、再生成を要求 |
| ネットワークエラー | リトライ後、オフラインモードの案内 |

---

## 9. 評価指標

### 9.1 定量指標

- タスク完了率（ユーザーの指示を正しく完了できた割合）
- 自律修正率（エラー発生後に自力で修正できた割合）
- 平均ステップ数（タスク完了までのツール呼び出し回数）
- トークン消費量・API コスト

### 9.2 定性指標

- 生成コードの品質（可読性、保守性）
- ユーザーへの説明の適切さ
- 不要な操作の少なさ
