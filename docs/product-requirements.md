# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**MyAgent** - CLI AI コーディングエージェント

### プロダクトコンセプト

- **ターミナルに住む自律型AI開発者**: ユーザーは自然言語で指示するだけで、コードの理解・生成・修正・検証をエージェントが自律的に実行する
- **Plan-Execute-Criticループ**: タスク分解、ツール実行、結果評価の思考サイクルを自律的に繰り返し、複雑な開発タスクを段階的に達成する
- **マルチLLMプロバイダ**: OpenAI / Gemini を切替・フォールバック可能な構成で、障害耐性とコスト最適化を実現する

### プロダクトビジョン

ソフトウェア開発者がターミナル上で自然言語の指示だけで開発作業を完結できるCLI AIエージェントを提供する。
ファイル操作、コード検索、Git操作、テスト実行などの開発ツールをLLMが自律的に組み合わせ、人間の開発者と同等の思考プロセスで問題を解決する。
Claude Codeと同等の開発体験を、オープンなLLMプロバイダ（OpenAI / Gemini）上で実現し、プロバイダロックインからの解放と拡張性を両立する。

### 目的

- ソフトウェア開発の全工程（設計・実装・テスト・デバッグ・リファクタリング）をCLI上で自律的に支援する
- LLMを活用した自律的なタスク分解・実行・検証の思考ループを実装する
- 複数LLMプロバイダに対応し、障害時のフォールバックとコスト最適化を可能にする
- 安全性を確保しつつ、ユーザーの生産性を最大化する

## 成功指標（KPI）

### プライマリーKPI

- タスク完了率: 開発者が自然言語で指示したタスクの80%以上を、人間の介入なしに正しく完了できる（プロジェクト開始6ヶ月後）
- 自律修正率: ツール実行エラー発生後、エージェントが自力で修正・再実行して成功する割合が60%以上（プロジェクト開始6ヶ月後）

### セカンダリーKPI

- 平均ステップ数: 標準的なタスク（単一ファイルのバグ修正）を10ステップ以内で完了
- ストリーミング応答遅延: 最初のトークンが1秒以内に表示開始
- LLMフォールバック成功率: プライマリプロバイダ障害時、セカンダリへの切替が95%以上成功

**受け入れ条件**:
- [x] `myagent` コマンドで対話モード（REPL）が起動する
- [x] `myagent "指示内容"` でワンショット実行ができる
- [x] LLMの応答がストリーミング（トークン単位）でリアルタイム表示される
- [x] Markdownのコードブロック、テーブルが適切にレンダリングされる
- [x] ツール実行中はスピナー/プログレスが表示される
- [x] 入力履歴が保持され、上下キーで呼び出せる

#### F5: エージェント思考ループ（Plan-Execute-Critic）

**ユーザーストーリー**:
開発者として、複雑なタスクをエージェントに自律的に分解・実行・検証させるために、思考ループ機能が欲しい

**受け入れ条件**:
- [x] LangGraphのステートマシンで `Plan → Execute → Observe → Evaluate → [Complete | Retry | Replan]` のループを実装する（ReActパターンで実装）
- [x] Plannerがユーザーの指示をサブタスクに分解し、実行計画を生成する（agent/planner.py）
- [x] Executorが計画に従いツールを順次呼び出し、結果を収集する（agent/executor.py、確認フロー判定）
- [x] Criticが実行結果を評価し、成功/失敗/再試行を判断する（agent/critic.py）
- [x] 最大ループ回数（デフォルト25回）を超えたら強制終了する（デフォルト20回で実装）
- [x] 同一ツール呼び出しの繰り返し（無限ループ）を検知して停止する（Critic.detect_loop + graph.py統合）

**優先度**: P0（必須）

#### F6: ユーザー確認フロー

**ユーザーストーリー**:
開発者として、エージェントが重要な操作を行う前に確認を求めてくるために、確認フロー機能が欲しい

**受け入れ条件**:
- [x] ファイル書き込み・編集時に差分を表示し、確認を求める
- [ ] 破壊的操作（force push, reset --hard等）の前に警告・確認を行う（F7 Git ツール実装時に対応）
- [x] 確認レベルを3段階で設定できる:
  - `strict`: 全操作で確認
  - `normal`: 書き込み・破壊操作のみ確認（デフォルト）
  - `autonomous`: 確認なし（上級者向け）
- [x] 確認プロンプトで `y/n/diff` の選択肢を提供する

**優先度**: P0（必須）

### 将来的な機能（Post-MVP）

#### F11: セッション永続化

**ユーザーストーリー**:
開発者として、作業を途中で中断して翌日に同じ状態で再開するために、セッション永続化機能が欲しい

**受け入れ条件**:

_保存と復元_:
- [ ] REPLセッション終了時（`exit` / `Ctrl+D`）に、会話履歴（コンテキスト圧縮済み）・AgentState・作業ディレクトリを `~/.myagent/sessions/<session-id>.json` に自動保存する
- [ ] `myagent --resume` で直近のセッションを再開できる
- [ ] `myagent --resume <session-id>` で指定のセッションを再開できる
- [ ] 再開時に前回のセッション概要をユーザーに表示する（最終操作日時、変更したファイル数、最後の指示内容）
- [ ] 再開時にプロジェクトインデックス（F9）を再構築し、中断後のファイル変更を反映する

_管理CLI_:
- [ ] `myagent session list` でセッション一覧（ID・作成日時・最終操作日時・概要）を表示できる
- [ ] `myagent session delete <session-id>` で指定のセッションを削除できる
- [ ] `myagent session delete --all` で全セッションを一括削除できる

_ストレージとライフサイクル_:
- [ ] セッションファイルのパーミッションを 600（所有者のみ読み書き）に設定する
- [ ] 30日間アクセスのないセッションファイルを起動時に自動削除する
- [ ] セッションファイルの合計サイズが 50MB を超えた場合に古い順から警告を表示する

_エラーハンドリング_:
- [ ] セッションファイルが破損している場合にエラーメッセージを表示し、新規セッションにフォールバックする
- [ ] バージョンアップでセッションのスキーマが変わった場合に警告を表示し、新規セッションを開始する（古いファイルは手動削除まで保持）

**関連機能**: F9 コンテキスト管理（圧縮済み履歴の保存・復元）、F10 設定管理（セッション保存先の設定）

**優先度**: P2（できれば）

#### F12: マルチエージェント協調

**ユーザーストーリー**:
開発者として、複数ファイルにまたがる大規模なタスクをエージェントに高速に処理させるために、サブタスクの並列実行機能が欲しい

**受け入れ条件**:

_タスク分割と並列実行_:
- [ ] Planner（F5）がサブタスク間の依存関係を分析し、独立したサブタスクを自動的に並列実行対象として識別する
- [ ] 並列実行の開始時にユーザーに「N個のサブタスクを並列実行します」と通知する
- [ ] 依存関係のあるサブタスクは順序を守って逐次実行する
- [ ] 並列実行数の上限を `config.toml` の `[agent]` セクションで設定できる（デフォルト: 3、最大: 10）

_ワーカー管理_:
- [ ] 各ワーカーエージェントは独立した会話コンテキストを持ち、他のワーカーの中間状態を参照・干渉しない
- [ ] 複数ワーカーが同一ファイルを編集しようとした場合にコンフリクトを検知し、該当タスクのみ逐次実行にフォールバックする
- [ ] 各ワーカーのファイル書き込みは確認フロー（F6）を個別に適用する

_結果統合と表示_:
- [ ] オーケストレーターが全ワーカーの結果を統合し、変更ファイル一覧と各サブタスクの成否をまとめて表示する
- [ ] 並列実行中の各ワーカーの進捗をリアルタイムにステータス表示する（例: `[2/3完了] テスト修正中...`）
- [ ] いずれかのワーカーが失敗した場合に、失敗理由を表示し、成功したワーカーの成果は保持する

_コスト管理_:
- [ ] 並列実行時のトークン使用量をワーカー単位で追跡し、合計コストを表示する
- [ ] 同時API呼び出しがプロバイダのレートリミットを超えないよう、ワーカー間で呼び出し間隔を制御する

**関連機能**: F5 エージェント思考ループ（Plannerによるタスク分解・依存分析の拡張）、F6 ユーザー確認フロー（ワーカー単位の確認）、F9 コンテキスト管理（ワーカーごとのコンテキスト分離）

**優先度**: P2（できれば）

#### F13: MCP（Model Context Protocol）対応

**ユーザーストーリー**:
開発者として、データベース・Slack・GitHub等の外部サービスをエージェントから操作するために、MCPプロトコルでツールを拡張できる機能が欲しい

**受け入れ条件**:

_サーバー接続_:
- [ ] MCPサーバー（stdioトランスポート）に接続し、公開されたツール・リソースを自動検出できる
- [ ] Streamable HTTPトランスポートに対応し、リモートのMCPサーバーと通信できる
- [ ] 接続するMCPサーバーを `config.toml` の `[[mcp.servers]]` セクションで設定できる
  ```toml
  [[mcp.servers]]
  name = "github"
  transport = "stdio"
  command = "npx"
  args = ["-y", "@modelcontextprotocol/server-github"]
  env = { GITHUB_TOKEN = "${GITHUB_TOKEN}" }
  ```
- [ ] アプリケーション起動時に設定済みのMCPサーバーへ自動接続する

_ツール統合_:
- [ ] 検出したMCPツールを `ToolRegistry` に動的に登録し、エージェントのツール選択候補に含める
- [ ] MCPツールのJSON Schema（入力・出力定義）をLLMに正しく渡す
- [ ] MCPリソース（読み取り専用データ）にアクセスし、エージェントのコンテキストとして利用できる
- [ ] MCPツールの呼び出しは確認フロー（F6）の対象とし、`normal` レベル以上で実行前に確認を求める

_管理CLIと可観測性_:
- [ ] `myagent mcp list` で接続中のサーバー名・ステータス・提供ツール数を一覧表示できる
- [ ] `myagent mcp test <server-name>` でサーバーへの接続確認とツール一覧の取得ができる
- [ ] MCPツールの呼び出し履歴（入力・出力・所要時間）をセッションログに記録する

_エラーハンドリング_:
- [ ] MCPサーバーへの接続失敗時に警告を表示し、内蔵ツールのみでエージェントを起動する（グレースフルデグラデーション）
- [ ] セッション中にMCPサーバーが切断された場合に自動再接続を試み、失敗時はユーザーに通知してそのサーバーのツールを無効化する
- [ ] MCPツール実行のタイムアウトをサーバーごとに設定できる（デフォルト: 30秒）

_セキュリティ_:
- [ ] MCPサーバーの環境変数に設定したシークレット（APIトークン等）はログに `***` でマスキングする
- [ ] MCPツールによる外部サービスへのアクセスは内蔵ツールのプロジェクトディレクトリ制限（F3）とは独立だが、確認フロー（F6）によって安全性を担保する

**関連機能**: F3 ファイル操作ツール（ToolRegistryの共用）、F6 ユーザー確認フロー（MCPツールの確認制御）、F10 設定管理（MCPサーバー設定）

**優先度**: P2（できれば）

#### F14: スキル拡張（Agent Skills Standard 準拠）

**ユーザーストーリー**:
開発者として、プロジェクト固有の定型ワークフロー（コードレビュー、リリース手順、テスト戦略等）をスキルとして定義し、またコミュニティが公開するスキル（Agent Skills Standard 準拠）をインストールして、エージェントに再利用可能な専門知識を持たせるために、スキル拡張機能が欲しい

**受け入れ条件**:

_スキル形式（Agent Skills Standard 準拠）_:
- [ ] スキルは `<skill-name>/SKILL.md` ディレクトリ構成で定義する
  ```
  code-review/
  ├── SKILL.md          # 必須: YAMLフロントマター + Markdown指示
  ├── scripts/          # 任意: 実行可能なスクリプト
  ├── references/       # 任意: 参照ドキュメント
  └── assets/           # 任意: テンプレート、リソース
  ```
- [ ] `SKILL.md` は YAML フロントマター + Markdown ボディで構成される
  ```markdown
  ---
  name: code-review
  description: >
  ```

- [ ] プロジェクトローカルスキルを `.myagent/skills/<skill-name>/SKILL.md` に配置できる

- [ ] スキルがアクティベートされた時点で `SKILL.md` ボディ全体をロードする
- [ ] `scripts/`、`references/`、`assets/` 内のファイルはスキル実行中に必要に応じてオンデマンドで読み込む
- [ ] 同名スキルはプロジェクトローカルが優先する（オーバーライド）

_スキルのアクティベーションと実行_:
- [ ] エージェントがユーザーの指示内容とスキルの `description` を照合し、関連するスキルを自動的にアクティベートする
- [ ] `/skill-name` の形式でスキルを明示的に呼び出すこともできる
- [ ] アクティベート時に `SKILL.md` のボディ（指示内容）がエージェントのコンテキストに注入される
- [ ] `allowed-tools` で指定されたツールはスキル実行中に確認なしで使用できる
- [ ] スキル実行中はスキル名を表示し、通常の会話と区別できる
- [ ] スキルの実行結果は会話コンテキスト（F9）に統合される

- [ ] `myagent skill install <path>` でローカルディレクトリからスキルをコピーできる
- [ ] Agent Skills Standard 準拠のスキルであれば、他のエージェント（Claude Code 等）向けに作成されたスキルもインストール・利用できる

_管理CLI_:
- [ ] `myagent skill list` でロード済みスキル一覧（名前・説明・スコープ・互換性）を表示できる
- [ ] `myagent skill info <skill-name>` でスキルの詳細（フロントマター全フィールド・ボディ概要・ファイル構成）を表示できる
- [ ] `myagent skill init <skill-name>` で Agent Skills Standard 準拠のスキル雛形を生成できる
- [ ] `myagent skill validate <path>` でスキルの `SKILL.md` フロントマターと命名規則を検証できる
- [ ] `myagent skill uninstall <skill-name>` でインストール済みスキルを削除できる

_エラーハンドリング_:

- [ ] `name` の命名規則違反（大文字、連続ハイフン等）を検出し、警告を表示する
- [ ] `SKILL.md` のパースエラー（不正なYAML等）時にスキップし、他のスキルのロードに影響しない

#### F15: プラグイン拡張

**ユーザーストーリー**:
開発者として、コミュニティや社内で開発されたプラグインパッケージをインストールして、エージェントにカスタムツール・プロバイダ・フォーマッタを追加するために、プラグイン拡張機能が欲しい

**受け入れ条件**:

_プラグインのインストールと管理_:
- [ ] `myagent plugin install <package-name>` で PyPI または Git リポジトリからプラグインをインストールできる
- [ ] `myagent plugin uninstall <package-name>` でプラグインをアンインストールできる
- [ ] `myagent plugin list` でインストール済みプラグイン一覧（名前・バージョン・提供機能）を表示できる
- [ ] `myagent plugin update <package-name>` でプラグインを最新バージョンに更新できる
- [ ] プラグインは `~/.myagent/plugins/` にインストールされる

_プラグインインターフェース_:
- [ ] プラグインは Python パッケージとして配布され、`myagent.plugins` エントリポイントで自動検出される
  ```toml
  # プラグインの pyproject.toml
  [project.entry-points."myagent.plugins"]
  my-plugin = "myagent_my_plugin:register"
  ```
- [ ] プラグインが提供できる拡張ポイント:
  - カスタムツール（`BaseTool` 準拠）→ `ToolRegistry` に動的登録
  - カスタム出力フォーマッタ → CLI表示のカスタマイズ
  - カスタムイベントハンドラ → エージェントイベントへのフック
- [ ] プラグインの `register()` 関数がアプリケーション起動時に呼ばれ、拡張を登録する
- [ ] プラグインごとの設定を `config.toml` の `[plugins.<plugin-name>]` セクションで管理できる

_安全性と分離_:
- [ ] プラグインが提供するツールは確認フロー（F6）の対象とする
- [ ] プラグインのロードエラー（import失敗、非互換バージョン等）は警告ログを出力し、他のプラグインや本体の動作に影響しない
- [ ] プラグインが対応するMyAgentのバージョン範囲を `plugin.toml` または `pyproject.toml` で宣言でき、非互換バージョンではロードをスキップする
- [ ] プラグインのツール実行はサンドボックスの制約（F3のディレクトリ制限、F4のコマンドブロックリスト）を継承する

_エラーハンドリング_:
- [ ] インストール時にパッケージが見つからない場合に適切なエラーメッセージを表示する
- [ ] プラグインの初期化エラー時にスタックトレースをログに記録し、該当プラグインのみ無効化する
- [ ] 起動時にプラグインのバージョン非互換を検出した場合に警告を表示し、`myagent plugin update` を提案する

**関連機能**: F3 ファイル操作ツール（ToolRegistryの共用・サンドボックス継承）、F6 ユーザー確認フロー（プラグインツールの確認制御）、F10 設定管理（プラグイン設定）、F13 MCP対応（外部ツール統合の棲み分け）

**優先度**: P2（できれば）

#### F16: カスタムコマンド

**ユーザーストーリー**:
開発者として、よく使う操作や定型処理をスラッシュコマンドとして登録し、短い入力で繰り返し実行するために、カスタムコマンド機能が欲しい

**受け入れ条件**:

_コマンド定義_:
- [ ] プロジェクトローカルコマンドを `.myagent/commands/<command-name>.toml` に定義できる
- [ ] グローバルコマンドを `~/.myagent/commands/<command-name>.toml` に定義できる
- [ ] コマンド定義ファイルの形式:
  ```toml
  # .myagent/commands/test-fix.toml
  name = "test-fix"
  description = "テストを実行し、失敗を自動修正する"
  # prompt: エージェントに送信するプロンプトテンプレート
  prompt = """
  以下の手順で作業してください:
  1. `{{test_command}}` を実行
  2. 失敗したテストがあれば原因を分析
  3. コードを修正して再度テストを実行
  4. 全テストがパスするまで繰り返す
  """
  [arguments]
  test_command = { description = "テスト実行コマンド", default = "pytest" }
  ```
- [ ] 同名コマンドはプロジェクトローカルが優先する（オーバーライド）

_コマンド実行_:
- [ ] REPL内で `/<command-name> [args]` の形式でコマンドを呼び出せる（例: `/test-fix --test_command "pytest -x"`）
- [ ] ワンショットモードで `myagent --command <command-name> [args]` として実行できる
- [ ] テンプレート内の `{{variable}}` がコマンド引数またはデフォルト値で置換される
- [ ] コマンド実行は通常のエージェント指示と同等に処理される（思考ループ・ツール使用・確認フロー全て適用）

_組み込みコマンド_:
- [ ] `/help` — コマンド一覧と使い方を表示
- [ ] `/clear` — 会話履歴をクリアする
- [ ] `/status` — 現在のセッション状態（トークン使用量、ループ回数、接続中のMCPサーバー等）を表示

_管理CLI_:
- [ ] `myagent command list` でロード済みコマンド一覧（名前・説明・スコープ・引数）を表示できる
- [ ] `myagent command init <command-name>` でコマンド定義の雛形ファイルを生成できる

_エラーハンドリング_:
- [ ] 未定義のコマンド名を入力した場合に「コマンドが見つかりません」と表示し、類似コマンド名を提案する
- [ ] コマンド定義ファイルのパースエラー時にスキップし、警告ログを出力する
- [ ] 必須引数が未指定の場合にエラーメッセージで引数定義を表示する

**関連機能**: F1 対話型CLI（REPL内でのスラッシュコマンド実行）、F5 エージェント思考ループ（コマンドのプロンプトをエージェント指示として処理）、F14 スキル拡張（スキルとコマンドの使い分け: スキルは複雑なワークフロー定義、コマンドは単一プロンプトのショートカット）

**優先度**: P2（できれば）

## 非機能要件

### パフォーマンス

- ストリーミング応答の最初のトークンが1秒以内に表示開始する
- 各ツール呼び出しは個別のタイムアウト制御を持つ（シェル: 120秒、ファイル: 10秒）
- 独立したツール呼び出しは並列実行可能とする

### ユーザビリティ

- 初回起動時に `myagent --help` で全機能の概要を確認できる
- エージェントの思考過程（どのツールをなぜ使ったか）がログで確認できる
- エラー発生時にユーザーが理解できるメッセージを表示する

### 信頼性

- LLM APIエラー時に指数バックオフリトライ（最大3回）を行う
- プロバイダ間フォールバックにより単一プロバイダ障害でもサービス継続する
- 自律ループの最大ステップ数を制限し（デフォルト25）、暴走を防止する
- 同一操作の無限ループを検知して自動停止する

### セキュリティ

- 危険コマンド（`rm -rf /`, `mkfs`, `dd` 等）をブロックリストで事前検知する
- ファイルアクセスをプロジェクトディレクトリ内に制限する
- APIキーは環境変数または `.env` ファイルから読み込み、ログに出力しない
- `.env`、`credentials` 等の機密ファイル読み取り時に警告を表示する

### ログ・オブザーバビリティ

- 全エージェント思考ログをJSON形式で `~/.myagent/logs/` に保存する
- ツール呼び出し履歴（入力・出力・所要時間）を記録する
- トークン使用量・APIコストをセッション単位で追跡する
- ログレベル（DEBUG / INFO / WARNING / ERROR）を設定可能とする

### スケーラビリティ

- LangChainのTool抽象に準拠し、新規ツールをクラス追加のみで拡張可能とする
- LLMプロバイダをインターフェース準拠で追加可能とする（将来のローカルLLM対応等）
- 設定ファイルで拡張ポイントを制御する

## スコープ外

明示的にスコープ外とする項目:

- GUI / Web UI の提供
- IDE統合（VSCode拡張等）
- リアルタイム共同編集
- マルチエージェント協調（Post-MVP: F12）
- スキル・プラグイン・コマンド拡張（Post-MVP: F14, F15, F16）
- ローカルLLMの直接サポート（将来拡張として検討）
- 自然言語以外の入力インターフェース（音声等）
