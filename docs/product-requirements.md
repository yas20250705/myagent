# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**MyAgent** - CLI AI コーディングエージェント

### プロダクトコンセプト

- **ターミナルに住む自律型AI開発者**: ユーザーは自然言語で指示するだけで、コードの理解・生成・修正・検証をエージェントが自律的に実行する
- **Plan-Execute-Criticループ**: タスク分解、ツール実行、結果評価の思考サイクルを自律的に繰り返し、複雑な開発タスクを段階的に達成する
- **マルチLLMプロバイダ**: OpenAI / Gemini を切替・フォールバック可能な構成で、障害耐性とコスト最適化を実現する

### プロダクトビジョン

ソフトウェア開発者がターミナル上で自然言語の指示だけで開発作業を完結できるCLI AIエージェントを提供する。
ファイル操作、コード検索、Git操作、テスト実行などの開発ツールをLLMが自律的に組み合わせ、人間の開発者と同等の思考プロセスで問題を解決する。
Claude Codeと同等の開発体験を、オープンなLLMプロバイダ（OpenAI / Gemini）上で実現し、プロバイダロックインからの解放と拡張性を両立する。

### 目的

- ソフトウェア開発の全工程（設計・実装・テスト・デバッグ・リファクタリング）をCLI上で自律的に支援する
- LLMを活用した自律的なタスク分解・実行・検証の思考ループを実装する
- 複数LLMプロバイダに対応し、障害時のフォールバックとコスト最適化を可能にする
- 安全性を確保しつつ、ユーザーの生産性を最大化する

## 成功指標（KPI）

### プライマリーKPI

- タスク完了率: 開発者が自然言語で指示したタスクの80%以上を、人間の介入なしに正しく完了できる（プロジェクト開始6ヶ月後）
- 自律修正率: ツール実行エラー発生後、エージェントが自力で修正・再実行して成功する割合が60%以上（プロジェクト開始6ヶ月後）

### セカンダリーKPI

- 平均ステップ数: 標準的なタスク（単一ファイルのバグ修正）を10ステップ以内で完了
- ストリーミング応答遅延: 最初のトークンが1秒以内に表示開始
- LLMフォールバック成功率: プライマリプロバイダ障害時、セカンダリへの切替が95%以上成功

## 機能要件

### コア機能（MVP）

#### F1: 対話型CLI（REPL）

**ユーザーストーリー**:
開発者として、ターミナル上でAIエージェントと自然言語で対話するために、対話型CLIインターフェースが欲しい

**受け入れ条件**:
- [ ] `myagent` コマンドで対話モード（REPL）が起動する
- [ ] `myagent "指示内容"` でワンショット実行ができる
- [ ] LLMの応答がストリーミング（トークン単位）でリアルタイム表示される
- [ ] Markdownのコードブロック、テーブルが適切にレンダリングされる
- [ ] ツール実行中はスピナー/プログレスが表示される
- [ ] 入力履歴が保持され、上下キーで呼び出せる
- [ ] `Ctrl+C` で現在の処理を中断できる

**優先度**: P0（必須）

---

#### F2: LLMプロバイダ管理（LLM Router）

**ユーザーストーリー**:
開発者として、OpenAIとGeminiを切り替えて使うために、プロバイダ管理機能が欲しい

**受け入れ条件**:
- [ ] OpenAI（GPT5-nano）とGemini（2.5 Pro, 2.5 Flash）に対応する
- [ ] 設定ファイルでデフォルトプロバイダ・モデルを指定できる
- [ ] `myagent config set model openai/gpt5-nano` のようにCLIから変更できる
- [ ] プライマリプロバイダのAPI障害時にフォールバックプロバイダへ自動切替する
- [ ] APIエラー時に指数バックオフでリトライする（最大3回）
- [ ] トークン使用量とAPI コストを追跡・表示できる

**優先度**: P0（必須）

---

#### F3: ファイル操作ツール

**ユーザーストーリー**:
開発者として、エージェントにプロジェクトのファイルを読み書き・検索させるために、ファイル操作ツール群が欲しい

**受け入れ条件**:
- [ ] `read_file`: ファイル全体または指定行範囲を読み込める
- [ ] `write_file`: 新規ファイルを作成できる
- [ ] `edit_file`: 既存ファイルの特定箇所を差分ベースで編集できる
- [ ] `list_directory`: ディレクトリ内容を一覧表示できる
- [ ] `glob_search`: globパターンでファイルを検索できる
- [ ] `grep_search`: 正規表現でコード内容を横断検索できる
- [ ] プロジェクトディレクトリ外へのファイル書き込みを禁止する

**優先度**: P0（必須）

---

#### F4: シェルコマンド実行ツール

**ユーザーストーリー**:
開発者として、エージェントにシェルコマンドを実行させるために、安全なコマンド実行機能が欲しい

**受け入れ条件**:
- [ ] `run_command`: 任意のシェルコマンドを実行し、stdout/stderrを取得できる
- [ ] タイムアウトを設定できる（デフォルト120秒）
- [ ] 危険コマンド（`rm -rf /`, `mkfs`, `dd` 等）をブロックリストで検知し、実行前に警告する
- [ ] 実行ディレクトリをプロジェクトルートに制限できる
- [ ] 作業ディレクトリ（cwd）の状態を追跡する

**優先度**: P0（必須）

---

#### F5: エージェント思考ループ（Plan-Execute-Critic）

**ユーザーストーリー**:
開発者として、複雑なタスクをエージェントに自律的に分解・実行・検証させるために、思考ループ機能が欲しい

**受け入れ条件**:
- [ ] LangGraphのステートマシンで `Plan → Execute → Observe → Evaluate → [Complete | Retry | Replan]` のループを実装する
- [ ] Plannerがユーザーの指示をサブタスクに分解し、実行計画を生成する
- [ ] Executorが計画に従いツールを順次呼び出し、結果を収集する
- [ ] Criticが実行結果を評価し、成功/失敗/再試行を判断する
- [ ] 最大ループ回数（デフォルト25回）を超えたら強制終了する
- [ ] 同一ツール呼び出しの繰り返し（無限ループ）を検知して停止する

**優先度**: P0（必須）

---

#### F6: ユーザー確認フロー

**ユーザーストーリー**:
開発者として、エージェントが重要な操作を行う前に確認を求めてくるために、確認フロー機能が欲しい

**受け入れ条件**:
- [ ] ファイル書き込み・編集時に差分を表示し、確認を求める
- [ ] 破壊的操作（force push, reset --hard等）の前に警告・確認を行う
- [ ] 確認レベルを3段階で設定できる:
  - `strict`: 全操作で確認
  - `normal`: 書き込み・破壊操作のみ確認（デフォルト）
  - `autonomous`: 確認なし（上級者向け）
- [ ] 確認プロンプトで `y/n/diff` の選択肢を提供する

**優先度**: P0（必須）

---

#### F7: Git操作ツール

**ユーザーストーリー**:
開発者として、エージェントにGit操作を実行させるために、Git統合ツールが欲しい

**受け入れ条件**:
- [ ] `git_status`: リポジトリの状態を確認できる
- [ ] `git_diff`: 変更差分を表示できる
- [ ] `git_log`: コミット履歴を表示できる
- [ ] `git_commit`: コミットメッセージを指定してコミットを作成できる
- [ ] `git_branch`: ブランチの作成・一覧表示ができる
- [ ] `git_checkout`: ブランチ切替・ファイル復元ができる
- [ ] 破壊的Git操作（force push, reset --hard, branch -D）は確認フローを経由する

**優先度**: P0（必須）

---

#### F8: テスト実行ツール

**ユーザーストーリー**:
開発者として、エージェントにテストを実行・解析させるために、テスト実行ツールが欲しい

**受け入れ条件**:
- [ ] `run_tests`: pytestを実行できる（ファイル単位、関数単位で指定可能）
- [ ] テスト結果（成功/失敗/エラー）を構造化して解析できる
- [ ] 失敗したテストの詳細（アサーションエラー、スタックトレース）をLLMに渡して修正を提案できる

**優先度**: P0（必須）

---

#### F9: コンテキスト管理

**ユーザーストーリー**:
開発者として、長い対話セッションでもエージェントがコンテキストを失わないために、コンテキスト管理機能が欲しい

**受け入れ条件**:
- [ ] 会話履歴のトークン使用量を常に追跡する
- [ ] コンテキストウィンドウの80%に達したら古い会話を要約して圧縮する
- [ ] ツール出力が大きい場合はトランケーションする
- [ ] プロジェクト初回ロード時にファイルツリーを自動取得・インデックス化する
- [ ] `.gitignore` に従ってインデックスから除外する

**優先度**: P1（重要）

---

#### F10: 設定管理

**ユーザーストーリー**:
開発者として、エージェントの動作をカスタマイズするために、設定管理機能が欲しい

**受け入れ条件**:
- [ ] `~/.myagent/config.toml` で設定を管理できる
- [ ] `myagent config set <key> <value>` で設定を変更できる
- [ ] `myagent config list` で現在の設定を一覧表示できる
- [ ] 以下の項目が設定可能:
  - LLMプロバイダ・モデル（デフォルト / フォールバック）
  - temperature、max_tokens
  - 最大イテレーション回数
  - 確認レベル（strict / normal / autonomous）
  - シェルタイムアウト
  - ログレベル・保存先

**優先度**: P1（重要）

### CLIインターフェース

```bash
# 対話モード（デフォルト）
myagent

# ワンショット実行
myagent "このバグを修正して"
myagent "テストを実行して失敗を修正して"

# 設定管理
myagent config set model openai/gpt5-nano
myagent config set model gemini/gemini-2.5-pro
myagent config set confirmation_level autonomous
myagent config list

# ヘルプ
myagent --help
```

## 非機能要件

### パフォーマンス

- ストリーミング応答の最初のトークンが1秒以内に表示開始する
- 各ツール呼び出しは個別のタイムアウト制御を持つ（シェル: 120秒、ファイル: 10秒）
- 独立したツール呼び出しは並列実行可能とする

### ユーザビリティ

- 初回起動時に `myagent --help` で全機能の概要を確認できる
- エージェントの思考過程（どのツールをなぜ使ったか）がログで確認できる
- エラー発生時にユーザーが理解できるメッセージを表示する

### 信頼性

- LLM APIエラー時に指数バックオフリトライ（最大3回）を行う
- プロバイダ間フォールバックにより単一プロバイダ障害でもサービス継続する
- 自律ループの最大ステップ数を制限し（デフォルト25）、暴走を防止する
- 同一操作の無限ループを検知して自動停止する

### セキュリティ

- 危険コマンド（`rm -rf /`, `mkfs`, `dd` 等）をブロックリストで事前検知する
- ファイルアクセスをプロジェクトディレクトリ内に制限する
- APIキーは環境変数または `.env` ファイルから読み込み、ログに出力しない
- `.env`、`credentials` 等の機密ファイル読み取り時に警告を表示する

### ログ・オブザーバビリティ

- 全エージェント思考ログをJSON形式で `~/.myagent/logs/` に保存する
- ツール呼び出し履歴（入力・出力・所要時間）を記録する
- トークン使用量・APIコストをセッション単位で追跡する
- ログレベル（DEBUG / INFO / WARNING / ERROR）を設定可能とする

### スケーラビリティ

- LangChainのTool抽象に準拠し、新規ツールをクラス追加のみで拡張可能とする
- LLMプロバイダをインターフェース準拠で追加可能とする（将来のローカルLLM対応等）
- 設定ファイルで拡張ポイントを制御する

## スコープ外

明示的にスコープ外とする項目:

- GUI / Web UI の提供
- IDE統合（VSCode拡張等）
- リアルタイム共同編集
- マルチエージェント協調（Post-MVP）
- ローカルLLMの直接サポート（将来拡張として検討）
- 自然言語以外の入力インターフェース（音声等）
