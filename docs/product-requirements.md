# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**MyAgent** - CLI AI コーディングエージェント

### プロダクトコンセプト

- **ターミナルに住む自律型AI開発者**: ユーザーは自然言語で指示するだけで、コードの理解・生成・修正・検証をエージェントが自律的に実行する
- **Plan-Execute-Criticループ**: タスク分解、ツール実行、結果評価の思考サイクルを自律的に繰り返し、複雑な開発タスクを段階的に達成する
- **マルチLLMプロバイダ**: OpenAI / Gemini を切替・フォールバック可能な構成で、障害耐性とコスト最適化を実現する

### プロダクトビジョン

ソフトウェア開発者がターミナル上で自然言語の指示だけで開発作業を完結できるCLI AIエージェントを提供する。
ファイル操作、コード検索、Git操作、テスト実行などの開発ツールをLLMが自律的に組み合わせ、人間の開発者と同等の思考プロセスで問題を解決する。
Claude Codeと同等の開発体験を、オープンなLLMプロバイダ（OpenAI / Gemini）上で実現し、プロバイダロックインからの解放と拡張性を両立する。

### 目的

- ソフトウェア開発の全工程（設計・実装・テスト・デバッグ・リファクタリング）をCLI上で自律的に支援する
- LLMを活用した自律的なタスク分解・実行・検証の思考ループを実装する
- 複数LLMプロバイダに対応し、障害時のフォールバックとコスト最適化を可能にする
- 安全性を確保しつつ、ユーザーの生産性を最大化する

## 成功指標（KPI）

### プライマリーKPI

- タスク完了率: 開発者が自然言語で指示したタスクの80%以上を、人間の介入なしに正しく完了できる（プロジェクト開始6ヶ月後）
- 自律修正率: ツール実行エラー発生後、エージェントが自力で修正・再実行して成功する割合が60%以上（プロジェクト開始6ヶ月後）

### セカンダリーKPI

- 平均ステップ数: 標準的なタスク（単一ファイルのバグ修正）を10ステップ以内で完了
- ストリーミング応答遅延: 最初のトークンが1秒以内に表示開始
- LLMフォールバック成功率: プライマリプロバイダ障害時、セカンダリへの切替が95%以上成功

**受け入れ条件**:
- [x] `myagent` コマンドで対話モード（REPL）が起動する
- [x] `myagent "指示内容"` でワンショット実行ができる
- [x] LLMの応答がストリーミング（トークン単位）でリアルタイム表示される
- [x] Markdownのコードブロック、テーブルが適切にレンダリングされる
- [x] ツール実行中はスピナー/プログレスが表示される
- [x] 入力履歴が保持され、上下キーで呼び出せる

#### F5: エージェント思考ループ（Plan-Execute-Critic）

**ユーザーストーリー**:
開発者として、複雑なタスクをエージェントに自律的に分解・実行・検証させるために、思考ループ機能が欲しい

**受け入れ条件**:
- [x] LangGraphのステートマシンで `Plan → Execute → Observe → Evaluate → [Complete | Retry | Replan]` のループを実装する（ReActパターンで実装）
- [x] Plannerがユーザーの指示をサブタスクに分解し、実行計画を生成する（agent/planner.py）
- [x] Executorが計画に従いツールを順次呼び出し、結果を収集する（agent/executor.py、確認フロー判定）
- [x] Criticが実行結果を評価し、成功/失敗/再試行を判断する（agent/critic.py）
- [x] 最大ループ回数（デフォルト25回）を超えたら強制終了する（デフォルト20回で実装）
- [x] 同一ツール呼び出しの繰り返し（無限ループ）を検知して停止する（Critic.detect_loop + graph.py統合）

**優先度**: P0（必須）

#### F6: ユーザー確認フロー

**ユーザーストーリー**:
開発者として、エージェントが重要な操作を行う前に確認を求めてくるために、確認フロー機能が欲しい

**受け入れ条件**:
- [x] ファイル書き込み・編集時に差分を表示し、確認を求める
- [ ] 破壊的操作（force push, reset --hard等）の前に警告・確認を行う（F7 Git ツール実装時に対応）
- [x] 確認レベルを3段階で設定できる:
  - `strict`: 全操作で確認
  - `normal`: 書き込み・破壊操作のみ確認（デフォルト）
  - `autonomous`: 確認なし（上級者向け）
- [x] 確認プロンプトで `y/n/diff` の選択肢を提供する

**優先度**: P0（必須）

### 将来的な機能（Post-MVP）

#### F11: セッション永続化

**ユーザーストーリー**:
開発者として、作業を途中で中断して翌日に同じ状態で再開するために、セッション永続化機能が欲しい

**受け入れ条件**:

_保存と復元_:
- [ ] REPLセッション終了時（`exit` / `Ctrl+D`）に、会話履歴（コンテキスト圧縮済み）・AgentState・作業ディレクトリを `~/.myagent/sessions/<session-id>.json` に自動保存する
- [ ] `myagent --resume` で直近のセッションを再開できる
- [ ] `myagent --resume <session-id>` で指定のセッションを再開できる
- [ ] 再開時に前回のセッション概要をユーザーに表示する（最終操作日時、変更したファイル数、最後の指示内容）
- [ ] 再開時にプロジェクトインデックス（F9）を再構築し、中断後のファイル変更を反映する

_管理CLI_:
- [ ] `myagent session list` でセッション一覧（ID・作成日時・最終操作日時・概要）を表示できる
- [ ] `myagent session delete <session-id>` で指定のセッションを削除できる
- [ ] `myagent session delete --all` で全セッションを一括削除できる

_ストレージとライフサイクル_:
- [ ] セッションファイルのパーミッションを 600（所有者のみ読み書き）に設定する
- [ ] 30日間アクセスのないセッションファイルを起動時に自動削除する
- [ ] セッションファイルの合計サイズが 50MB を超えた場合に古い順から警告を表示する

_エラーハンドリング_:
- [ ] セッションファイルが破損している場合にエラーメッセージを表示し、新規セッションにフォールバックする
- [ ] バージョンアップでセッションのスキーマが変わった場合に警告を表示し、新規セッションを開始する（古いファイルは手動削除まで保持）

**関連機能**: F9 コンテキスト管理（圧縮済み履歴の保存・復元）、F10 設定管理（セッション保存先の設定）

**優先度**: P2（できれば）

#### F12: マルチエージェント協調

**ユーザーストーリー**:
開発者として、複数ファイルにまたがる大規模なタスクをエージェントに高速に処理させるために、サブタスクの並列実行機能が欲しい

**受け入れ条件**:

_タスク分割と並列実行_:
- [x] Planner（F5）がサブタスク間の依存関係を分析し、独立したサブタスクを自動的に並列実行対象として識別する
- [x] 並列実行の開始時にユーザーに「N個のサブタスクを並列実行します」と通知する
- [x] 依存関係のあるサブタスクは順序を守って逐次実行する
- [x] 並列実行数の上限を `config.toml` の `[agent]` セクションで設定できる（デフォルト: 3、最大: 10）

_ワーカー管理_:
- [x] 各ワーカーエージェントは独立した会話コンテキストを持ち、他のワーカーの中間状態を参照・干渉しない
- [x] 複数ワーカーが同一ファイルを編集しようとした場合にコンフリクトを検知し、該当タスクのみ逐次実行にフォールバックする
- [x] 各ワーカーのファイル書き込みは確認フロー（F6）を個別に適用する

_結果統合と表示_:
- [x] オーケストレーターが全ワーカーの結果を統合し、変更ファイル一覧と各サブタスクの成否をまとめて表示する
- [x] 並列実行中の各ワーカーの進捗をリアルタイムにステータス表示する（例: `[2/3完了] テスト修正中...`）
- [x] いずれかのワーカーが失敗した場合に、失敗理由を表示し、成功したワーカーの成果は保持する

_コスト管理_:
- [x] 並列実行時のトークン使用量をワーカー単位で追跡し、合計コストを表示する
- [x] 同時API呼び出しがプロバイダのレートリミットを超えないよう、ワーカー間で呼び出し間隔を制御する

**関連機能**: F5 エージェント思考ループ（Plannerによるタスク分解・依存分析の拡張）、F6 ユーザー確認フロー（ワーカー単位の確認）、F9 コンテキスト管理（ワーカーごとのコンテキスト分離）

**優先度**: P2（できれば）

#### F13: MCP（Model Context Protocol）対応

**ユーザーストーリー**:
開発者として、データベース・Slack・GitHub等の外部サービスをエージェントから操作するために、MCPプロトコルでツールを拡張できる機能が欲しい

**受け入れ条件**:

_サーバー接続_:
- [x] MCPサーバー（stdioトランスポート）に接続し、公開されたツール・リソースを自動検出できる
- [x] Streamable HTTPトランスポートに対応し、リモートのMCPサーバーと通信できる
- [x] 接続するMCPサーバーを `config.toml` の `[[mcp.servers]]` セクションで設定できる
  ```toml
  [[mcp.servers]]
  name = "github"
  transport = "stdio"
  command = "npx"
  args = ["-y", "@modelcontextprotocol/server-github"]
  env = { GITHUB_TOKEN = "${GITHUB_TOKEN}" }
  ```
- [x] アプリケーション起動時に設定済みのMCPサーバーへ自動接続する

_ツール統合_:
- [x] 検出したMCPツールを `ToolRegistry` に動的に登録し、エージェントのツール選択候補に含める
- [x] MCPツールのJSON Schema（入力・出力定義）をLLMに正しく渡す
- [ ] MCPリソース（読み取り専用データ）にアクセスし、エージェントのコンテキストとして利用できる（基本実装のみ）
- [x] MCPツールの呼び出しは確認フロー（F6）の対象とし、`normal` レベル以上で実行前に確認を求める

_管理CLIと可観測性_:
- [x] `myagent mcp list` で接続中のサーバー名・ステータス・提供ツール数を一覧表示できる
- [x] `myagent mcp test <server-name>` でサーバーへの接続確認とツール一覧の取得ができる
- [ ] MCPツールの呼び出し履歴（入力・出力・所要時間）をセッションログに記録する

_エラーハンドリング_:
- [x] MCPサーバーへの接続失敗時に警告を表示し、内蔵ツールのみでエージェントを起動する（グレースフルデグラデーション）
- [x] セッション中にMCPサーバーが切断された場合に自動再接続を試み、失敗時はユーザーに通知してそのサーバーのツールを無効化する
- [x] MCPツール実行のタイムアウトをサーバーごとに設定できる（デフォルト: 30秒）

_セキュリティ_:
- [x] MCPサーバーの環境変数に設定したシークレット（APIトークン等）はログに `***` でマスキングする
- [x] MCPツールによる外部サービスへのアクセスは内蔵ツールのプロジェクトディレクトリ制限（F3）とは独立だが、確認フロー（F6）によって安全性を担保する

**関連機能**: F3 ファイル操作ツール（ToolRegistryの共用）、F6 ユーザー確認フロー（MCPツールの確認制御）、F10 設定管理（MCPサーバー設定）

**優先度**: P2（できれば）

#### F14: スキル拡張（Agent Skills Standard 準拠）

**ユーザーストーリー**:
開発者として、プロジェクト固有の定型ワークフロー（コードレビュー、リリース手順、テスト戦略等）をスキルとして定義し、またコミュニティが公開するスキル（Agent Skills Standard 準拠）をインストールして、エージェントに再利用可能な専門知識を持たせるために、スキル拡張機能が欲しい

**受け入れ条件**:

_スキル形式（Agent Skills Standard 準拠）_:
- [x] スキルは `<skill-name>/SKILL.md` ディレクトリ構成で定義する
  ```
  code-review/
  ├── SKILL.md          # 必須: YAMLフロントマター + Markdown指示
  ├── scripts/          # 任意: 実行可能なスクリプト
  ├── references/       # 任意: 参照ドキュメント
  └── assets/           # 任意: テンプレート、リソース
  ```
- [x] `SKILL.md` は YAML フロントマター + Markdown ボディで構成される
  ```markdown
  ---
  name: code-review
  description: >
    プルリクエストのコードレビューを実行する。

  1. 対象のファイルまたはブランチの差分を取得
  2. コード品質・セキュリティ・パフォーマンスの観点で分析

  ...
  ```
- [x] フロントマターの必須フィールド: `name`（小文字英数字+ハイフン、最大64文字）、`description`（最大1024文字）
- [x] フロントマターのオプションフィールド: `license`、`compatibility`（環境要件）、`metadata`（任意キーバリュー）、`allowed-tools`（事前承認ツールリスト）
- [x] `name` はディレクトリ名と一致しなければならない

_スキルの配置とロード_:
- [x] プロジェクトローカルスキルを `.myagent/skills/<skill-name>/SKILL.md` に配置できる
- [x] グローバルスキルを `~/.myagent/skills/<skill-name>/SKILL.md` に配置できる
- [x] アプリケーション起動時にプロジェクトローカル → グローバルの順にスキルを自動検出する
- [x] 起動時はメタデータ（`name` + `description`、約100トークン）のみをロードする（Progressive Disclosure）
- [x] スキルがアクティベートされた時点で `SKILL.md` ボディ全体をロードする
- [ ] `scripts/`、`references/`、`assets/` 内のファイルはスキル実行中に必要に応じてオンデマンドで読み込む
- [x] 同名スキルはプロジェクトローカルが優先する（オーバーライド）

_スキルのアクティベーションと実行_:
- [x] エージェントがユーザーの指示内容とスキルの `description` を照合し、関連するスキルを自動的にアクティベートする
- [x] `/skill-name` の形式でスキルを明示的に呼び出すこともできる
- [x] アクティベート時に `SKILL.md` のボディ（指示内容）がエージェントのコンテキストに注入される
- [ ] `allowed-tools` で指定されたツールはスキル実行中に確認なしで使用できる
- [x] スキル実行中はスキル名を表示し、通常の会話と区別できる
- [x] スキルの実行結果は会話コンテキスト（F9）に統合される

_スキルのインストール_:
- [x] `myagent skill install <git-repo-url>` で Git リポジトリからスキルをインストールできる
- [x] `myagent skill install <path>` でローカルディレクトリからスキルをコピーできる
- [x] Agent Skills Standard 準拠のスキルであれば、他のエージェント（Claude Code 等）向けに作成されたスキルもインストール・利用できる

_管理CLI_:
- [x] `myagent skill list` でロード済みスキル一覧（名前・説明・スコープ・互換性）を表示できる
- [x] `myagent skill info <skill-name>` でスキルの詳細（フロントマター全フィールド・ボディ概要・ファイル構成）を表示できる
- [x] `myagent skill init <skill-name>` で Agent Skills Standard 準拠のスキル雛形を生成できる
- [x] `myagent skill validate <path>` でスキルの `SKILL.md` フロントマターと命名規則を検証できる

_エラーハンドリング_:
- [x] `SKILL.md` の必須フィールド（`name`、`description`）欠落時にスキップし、警告ログを出力する
- [x] `name` がディレクトリ名と不一致の場合にスキップし、警告ログを出力する

#### F15: プラグイン拡張

**ユーザーストーリー**:
開発者として、コミュニティや社内で開発されたプラグインをインストールして、エージェントにカスタムスキル・フック・MCPサーバー・エージェント定義を追加するために、Claude Code Plugins Standard 準拠のプラグイン拡張機能が欲しい

**受け入れ条件**:

_プラグインディレクトリ構造_:
- [x] プラグインは以下のディレクトリ構造に準拠する
  ```
  my-plugin/
    .claude-plugin/
      plugin.json           # マニフェスト（name 必須）
    skills/                 # スキル定義（SKILL.md）
      skill-name/
        SKILL.md
    commands/               # レガシーコマンド（*.md）
    agents/                 # サブエージェント定義（*.md）
    hooks/
      hooks.json            # イベントフック定義
    .mcp.json               # MCPサーバー定義
    settings.json           # プラグイン設定（agent キーのみ）
  ```
- [x] `.claude-plugin/plugin.json` にはマニフェストのみを配置し、他のコンポーネントはプラグインルートに配置する
- [x] `plugin.json` が省略された場合、コンポーネントをデフォルトパスから自動検出し、プラグイン名はディレクトリ名から導出する

_プラグインマニフェスト（plugin.json）_:
- [x] `name`（kebab-case）を必須フィールドとする
- [x] オプションフィールド: `version`（semver）、`description`、`author`（name/email/url）、`homepage`、`repository`、`license`、`keywords`
- [x] カスタムパス指定: `skills`、`commands`、`agents`、`hooks`、`mcpServers` にプラグインルートからの相対パス（`./` 始まり）を指定できる
- [x] カスタムパスはデフォルトディレクトリを置換せず補足する

_プラグインが提供できる拡張ポイント_:
- [x] **スキル**: `skills/<name>/SKILL.md`（YAMLフロントマター + Markdownボディ、F14のスキル形式と同一）

- [x] **フック**: `hooks/hooks.json` でイベント（PreToolUse, PostToolUse 等）に対するコマンド実行を定義できる
- [x] **MCPサーバー**: `.mcp.json` でローカル（stdio）またはリモート（HTTP）のMCPサーバーを定義でき、プラグイン有効時に自動起動する
- [ ] スキルは `/plugin-name:skill-name` 形式で名前空間が分離される（モデル上は設計済み、実行時統合は未実装）

_環境変数_:
- [ ] `${MYAGENT_PLUGIN_ROOT}`: プラグインインストールディレクトリの絶対パス（スキル内容・フックコマンド・MCP設定で利用可能）
- [ ] `${MYAGENT_PLUGIN_DATA}`: プラグインごとの永続データディレクトリ（`~/.myagent/plugins/data/<id>/`、更新後も維持、アンインストール時に削除）

_インストールと管理_:
- [x] `myagent plugin install <source>` でローカルパスまたは Git リポジトリからプラグインをインストールできる
- [x] `myagent plugin uninstall <name>` でプラグインをアンインストールできる（`--keep-data` でデータ保持可能）
- [x] `myagent plugin list` でインストール済みプラグイン一覧（名前・バージョン・提供コンポーネント）を表示できる
- [x] `myagent plugin update <name>` でプラグインを最新バージョンに更新できる
- [x] `myagent plugin enable/disable <name>` でプラグインの有効・無効を切り替えられる
- [x] `myagent plugin validate [path]` でプラグインのマニフェストとコンポーネント構造を検証できる

_インストールスコープ_:
- [x] `--scope user`: ユーザーグローバル（`~/.myagent/settings.json`）、全プロジェクトで有効（CLIオプション実装済み）
- [ ] `--scope project`: プロジェクト共有（`.myagent/settings.json`）、VCS で共有（スコープ別インストール先の分岐は将来実装）
- [ ] `--scope local`: プロジェクトローカル（`.myagent/settings.local.json`）、gitignore 対象（同上）

_プラグインの検出とロード_:
- [x] インストール済みプラグインはプラグインキャッシュ（`~/.myagent/plugins/cache/`）にコピーして管理する
- [x] 起動時にメタデータ（plugin.json）のみロードし、コンポーネントはオンデマンドでロードする
- [ ] `--plugin-dir` オプションで開発中のプラグインをローカルディレクトリから直接ロードできる
- [ ] `/reload-plugins` コマンドで再起動なしにプラグインをリロードできる

_安全性と分離_:
- [ ] プラグインが提供するスキル・ツールは確認フロー（F6）の対象とする
- [x] プラグインはプラグインディレクトリ外のファイルを参照できない（パストラバーサル防止）
- [x] プラグインのロードエラーは警告ログを出力し、他のプラグインや本体の動作に影響しない
- [ ] フックの `PreToolUse` イベントでツール実行のブロックが可能

_エラーハンドリング_:
- [x] インストール時にソースが見つからない場合に適切なエラーメッセージを表示する
- [x] プラグインの初期化エラー時にログに記録し、該当プラグインのみ無効化する
- [ ] `--debug` フラグでプラグインロードの詳細情報を表示できる

**関連機能**: F3 ファイル操作ツール（サンドボックス継承）、F6 ユーザー確認フロー（プラグインツールの確認制御）、F10 設定管理（プラグイン設定）、F13 MCP対応（プラグインMCPサーバーとの統合）、F14 スキル拡張（スキル形式の共有）

**優先度**: P2（できれば）

#### F16: カスタムコマンド

**ユーザーストーリー**:
開発者として、よく使う操作や定型処理をスラッシュコマンドとして登録し、短い入力で繰り返し実行するために、カスタムコマンド機能が欲しい

_コマンド定義_:

- [x] ワンショットモードで `myagent --command <command-name> --command-args '--key val'` として実行できる
- [x] テンプレート内の `{{variable}}` がコマンド引数またはデフォルト値で置換される
- [x] コマンド実行は通常のエージェント指示と同等に処理される（思考ループ・ツール使用・確認フロー全て適用）

_組み込みコマンド_:
- [x] `/help` — コマンド一覧と使い方を表示
- [x] `/clear` — 会話履歴をクリアする
- [x] `/status` — 現在のセッション状態（トークン使用量、ループ回数等）を表示

_管理CLI_:
- [x] `myagent command list` でロード済みコマンド一覧（名前・説明・スコープ・引数）を表示できる
- [x] `myagent command init <command-name>` でコマンド定義の雛形ファイルを生成できる

_エラーハンドリング_:
- [x] 未定義のコマンド名を入力した場合に「コマンドが見つかりません」と表示し、類似コマンド名を提案する
- [x] コマンド定義ファイルのパースエラー時にスキップし、警告ログを出力する
- [x] 必須引数が未指定の場合にエラーメッセージで引数定義を表示する

**関連機能**: F1 対話型CLI（REPL内でのスラッシュコマンド実行）、F5 エージェント思考ループ（コマンドのプロンプトをエージェント指示として処理）、F14 スキル拡張（スキルとコマンドの使い分け: スキルは複雑なワークフロー定義、コマンドは単一プロンプトのショートカット）

**優先度**: P2（できれば）

#### F17: Web検索・Webページ取得

**ユーザーストーリー**:
開発者として、最新のライブラリドキュメント・APIリファレンス・技術記事等をエージェントがリアルタイムに検索・取得し、LLMの学習データにない最新情報に基づいて回答・実装を行えるようにするために、Web検索・Webページ取得機能が欲しい

**受け入れ条件**:

_Web検索ツール（`websearch`）_:
- [x] `websearch` ツールを `ToolRegistry` に登録し、エージェントのツール選択候補に含める

- [x] 検索バックエンドとして、Exa AI REST API を利用できる（MCPサーバー経由ではなく直接HTTP）
- [x] 検索バックエンドの設定を `config.toml` で管理できる（エンドポイントURL、APIキー環境変数名等）

_Webページ取得ツール（`webfetch`）_:
- [x] `webfetch` ツールを `ToolRegistry` に登録し、エージェントのツール選択候補に含める
- [x] URL を受け取り、ページ内容を取得して Markdown 形式に変換して返す
- [x] 出力形式を `format` パラメータで選択できる（`markdown` / `text` / `html`、デフォルト: `markdown`）

_エージェント統合_:

- [x] 取得したWeb情報をエージェントのコンテキスト（F9）に統合し、後続の思考ループで活用できる（既存ContextManagerが自動処理）

_確認フローと安全性_:
- [ ] `websearch` / `webfetch` ツールの実行は確認フロー（F6）の対象とし、`normal` レベル以上で実行前に確認を求める（将来拡張）
- [ ] 取得したWebコンテンツに含まれる可能性のあるプロンプトインジェクションに対して、ユーザーに警告を表示する仕組みを持つ（将来拡張）

_フォールバック検索_:
- [x] Exa AI バックエンドが失敗（API エラー、タイムアウト、認証失敗等）した場合に、代替の Web 検索バックエンドへ自動的にフォールバックする
- [x] フォールバック先として DuckDuckGo 等の API キー不要な検索バックエンドを利用できる
- [x] フォールバック発生時にログに警告を出力し、ユーザーにはフォールバックで取得した結果であることを明示する
- [x] フォールバックの有効/無効および優先順位を `config.toml` で設定できる
- [x] Exa API キーが未設定の場合でも、フォールバックバックエンドのみで `websearch` ツールが利用可能である

_エラーハンドリング_:
- [x] 検索バックエンドへの接続失敗時に適切なエラーメッセージを表示し、エージェントの他のツール使用に影響しない
- [x] レスポンスサイズ超過時にエラーメッセージを返す
- [x] Cloudflare等によるブロック時に適切なエラーメッセージを返す
- [x] タイムアウト時に `ToolExecutionError` を返す（MCPTimeoutErrorではなくToolExecutionErrorで統一）

**関連機能**: F3 ファイル操作ツール（ToolRegistryの共用）、F6 ユーザー確認フロー（Web系ツールの確認制御）、F9 コンテキスト管理（取得情報のコンテキスト統合）、F13 MCP対応（MCPサーバー経由の検索バックエンド利用）

**優先度**: P2（できれば）

#### F18: エージェント精度向上

**ユーザーストーリー**:
開発者として、エージェントがより正確で一貫性のある応答を返し、ツール選択の失敗や不要なループを減らすために、エージェントの推論・実行精度を向上させる機能が欲しい

**背景**:
現状のエージェントは基本的なReActパターンで動作するが、複雑なタスクにおいてツール選択の誤り、コンテキスト喪失による精度低下、出力の意味的検証不足などの課題がある。本機能は [ImprovementOfAIAgent.md](references/ImprovementOfAIAgent.md) の調査結果に基づき、段階的にエージェント精度を向上させる。

**受け入れ条件**:

_動的プロンプト最適化_:
- [x] タスク種別（コード生成、バグ修正、リファクタリング、調査等）に応じてシステムプロンプトのテンプレートを動的に切り替える
- [x] プロンプトテンプレートを外部ファイル（`prompts/` ディレクトリ）に配置し、コードから分離して管理できる
- [ ] LLMプロバイダ・モデル固有の最適化（トークン効率、指示形式等）をテンプレート単位で適用できる

_コンテキスト管理の高度化（F9拡張）_:
- [x] コンテキストメッセージに優先度レベル（critical / normal / low）を付与し、圧縮時に優先度に基づいて保持・削除を判断する
- [x] アーキテクチャ上の決定事項やユーザーの重要な制約条件を「固定コンテキスト（Inception Message）」として圧縮対象から除外できる
- [x] ツール実行結果のうち、後続の推論で不要になった中間出力（成功した読み取り結果の重複等）を動的に刈り込む（Dynamic Context Pruning）

_ツール選択精度の向上_:
- [ ] ユーザーの意図を分類（ファイル操作、検索、実行、Git操作等）し、分類結果に基づいて提示するツール候補を絞り込むルーターを実装する
- [ ] ツール定義のスキーマ（パラメータ説明・使用例・制約条件）を充実させ、LLMが正確にパラメータを生成できるようにする
- [x] ツール呼び出し前にパラメータのバリデーションを行い、不正な入力を検知した場合はLLMに再生成を要求する

_出力の意味的検証_:
- [ ] Critic（F5）がツール実行結果を意味的に検証し、期待される出力形式・内容との乖離を検知する
- [ ] ファイル編集後に構文チェック（該当言語のlinter/parser）を自動実行し、構文エラーを即座にフィードバックする
- [x] 同一エラーの繰り返し（3回以上同じツールが同じエラーで失敗）を検知した場合にアプローチの変更を促す

_エージェント評価メトリクス_:
- [x] セッションごとにツール呼び出し正答率（成功/失敗比）を計測・記録する

#### F19: REPL内CLIサブコマンド実行

**ユーザーストーリー**:
開発者として、myagentのREPLセッション内からスラッシュコマンド形式でCLI管理コマンド（plugin / skill / command / config / mcp 等）を実行できるようにしたい。現在はターミナルで `uv run myagent plugin list` のように別プロセスとして実行する必要があり、REPLを離れずに管理操作を行いたい。

**背景**:
現在、プラグイン管理・スキル管理・コマンド管理・設定変更・MCP接続確認などのCLI管理操作は、REPLの外でターミナルコマンドとして実行する必要がある。REPL内で `/plugin list` のようにスラッシュコマンドとして実行できれば、作業の中断なく管理操作が行える。

**受け入れ条件**:

_プラグイン管理（`/plugin`）_:
- [x] `/plugin list` でインストール済みプラグインの一覧を表示できる
- [x] `/plugin install <git-url-or-path>` でプラグインをインストールできる
- [x] `/plugin uninstall <name>` でプラグインをアンインストールできる
- [x] `/plugin enable <name>` でプラグインを有効化できる
- [x] `/plugin disable <name>` でプラグインを無効化できる
- [x] `/plugin validate [path]` でマニフェストを検証できる

_スキル管理（`/skill`）_:
- [x] `/skill list` でスキル一覧を表示できる
- [x] `/skill info <skill-name>` でスキルの詳細情報を表示できる
- [x] `/skill validate <path>` でSKILL.mdを検証できる

- [x] `/skill uninstall <skill-name>` でスキルを削除できる

- [x] `/command list` で登録済みカスタムコマンドの一覧を表示できる
- [x] `/command init <name>` でプロジェクトローカルのコマンド雛形を生成できる

_設定管理（`/config`, `/set-config`）_:
- [x] `/config` で現在の設定内容を表示できる
- [x] `/config` でグローバル設定とプロジェクトローカル設定のマージ結果が表示される（F20の設定階層マージを反映）

_共通要件_:
- [x] 各スラッシュコマンドの実行結果がREPL内にRich形式で整形表示される

#### F20: グローバルCLIインストールとカレントディレクトリ作業

**ユーザーストーリー**:
開発者として、任意のプロジェクトフォルダでPowerShell（またはターミナル）を開き、`myagent` とだけ入力すればエージェントが起動し、そのカレントディレクトリを作業フォルダとして自動認識してほしい。`uv run myagent` という前置きなしに、どのディレクトリからでもすぐに使い始めたい。

**背景**:
現在は `uv run myagent` でしか起動できず、myagentプロジェクトのディレクトリ内にいる必要がある。開発者が自分のプロジェクトフォルダで直接 `myagent` と入力して使えるようにすることで、日常的な利用が格段に容易になる。

**受け入れ条件**:

_グローバルインストール_:
- [x] `uv tool install` または `pip install` によりシステム全体で `myagent` コマンドが利用可能になる
- [x] インストール後、任意のディレクトリで `myagent` と入力するとREPLが起動する
- [x] `myagent --version` でバージョン情報を表示できる
- [x] インストール手順をMANUAL.mdに記載する

_カレントディレクトリの自動認識_:
- [x] 起動時のカレントディレクトリ（`cwd`）をエージェントの作業ディレクトリとして自動設定する
- [x] ファイル操作ツール（F3）がカレントディレクトリを基準に相対パスを解決する
- [x] シェル実行ツール（F4）がカレントディレクトリを作業ディレクトリとしてコマンドを実行する
- [x] `--working-dir <path>` オプションで作業ディレクトリを明示的に指定できる（カレントディレクトリより優先）
- [x] 起動時に作業ディレクトリのパスを表示する

_設定ファイルの解決_:
- [x] グローバル設定（`~/.myagent/config.toml`）がデフォルトで読み込まれる
- [x] カレントディレクトリに `.myagent/config.toml` が存在する場合、プロジェクトローカル設定としてグローバル設定にマージされる（プロジェクト設定が優先）
- [x] `--config <path>` オプションによる明示指定が最優先される

_カスタムコマンド・スキルの解決_:
- [x] カレントディレクトリの `.myagent/commands/` にあるカスタムコマンドが自動的にロードされる
- [x] カレントディレクトリの `.myagent/skills/` にあるスキルが自動的にロードされる
- [x] グローバル（`~/.myagent/commands/`, `~/.myagent/skills/`）とプロジェクトローカルの両方がロードされ、同名はプロジェクトローカルが優先される

_エラーハンドリング_:
- [x] ~~グローバルインストールされていない状態で `myagent` を実行した場合、インストール手順を案内するエラーメッセージを表示する~~（シェルレベルの "command not found" エラーのため、アプリ内では制御不可。MANUAL.mdにインストール手順を記載することで代替）
- [x] APIキーが未設定の場合、起動時に設定方法を案内する（`.env` またはシステム環境変数）
- [x] カレントディレクトリへの読み取り権限がない場合、適切なエラーメッセージを表示する

**関連機能**: F1 CLI基盤（Clickコマンド構造）、F3 ファイル操作ツール（作業ディレクトリの基準）、F4 シェル実行ツール（cwdの設定）、F8 設定管理（設定ファイルのマージ戦略）、F16 カスタムコマンド（コマンド探索パス）、F14 スキル拡張（スキル探索パス）

**優先度**: P1（必須）

## 非機能要件

### パフォーマンス

- ストリーミング応答の最初のトークンが1秒以内に表示開始する
- 各ツール呼び出しは個別のタイムアウト制御を持つ（シェル: 120秒、ファイル: 10秒）
- 独立したツール呼び出しは並列実行可能とする

### ユーザビリティ

- 初回起動時に `myagent --help` で全機能の概要を確認できる
- エージェントの思考過程（どのツールをなぜ使ったか）がログで確認できる
- エラー発生時にユーザーが理解できるメッセージを表示する

### 信頼性

- LLM APIエラー時に指数バックオフリトライ（最大3回）を行う
- プロバイダ間フォールバックにより単一プロバイダ障害でもサービス継続する
- 自律ループの最大ステップ数を制限し（デフォルト25）、暴走を防止する
- 同一操作の無限ループを検知して自動停止する

### セキュリティ

- 危険コマンド（`rm -rf /`, `mkfs`, `dd` 等）をブロックリストで事前検知する
- ファイルアクセスをプロジェクトディレクトリ内に制限する
- APIキーは環境変数または `.env` ファイルから読み込み、ログに出力しない
- `.env`、`credentials` 等の機密ファイル読み取り時に警告を表示する

### ログ・オブザーバビリティ

- 全エージェント思考ログをJSON形式で `~/.myagent/logs/` に保存する
- ツール呼び出し履歴（入力・出力・所要時間）を記録する
- トークン使用量・APIコストをセッション単位で追跡する
- ログレベル（DEBUG / INFO / WARNING / ERROR）を設定可能とする

### スケーラビリティ

- LangChainのTool抽象に準拠し、新規ツールをクラス追加のみで拡張可能とする
- LLMプロバイダをインターフェース準拠で追加可能とする（将来のローカルLLM対応等）
- 設定ファイルで拡張ポイントを制御する

- GUI / Web UI の提供
- IDE統合（VSCode拡張等）
- リアルタイム共同編集
- マルチエージェント協調（Post-MVP: F12）
- スキル・プラグイン・コマンド拡張（Post-MVP: F14, F15, F16）

- ローカルLLMの直接サポート（将来拡張として検討）
